# AGENTS.md — 灵镜 (LingPrism) AI 协作规范

> 项目代号：**Prism（棱镜墨斯）** · 产品名称：**灵镜 (LingPrism)**
>
> 本文档定义 AI 协作代理在本仓库中的角色、行为边界与交付标准。所有在本仓库工作的 AI Agent 必须遵循本文档。

---

## 项目概述

灵镜 (LingPrism) 是一个**企业项目与知识管理平台**，帮助员工了解公司内部项目架构、技术栈与知识资产，并为领导层提供全局架构视图、系统设计洞察及代码/安全风险感知。

**核心能力：**

- **员工视角**：浏览项目架构、技术栈、团队知识库
- **管理视角**：同步企业内部代码库、维护知识库、生成培训/技术文档
- **领导视角**：全局架构、系统设计、代码风险与安全风险监控
- **对外服务**：提供 MCP 服务，供其他 Agent 调用平台能力

## 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 代码解析与图谱构建 | Rust + tree-sitter | 参考开源项目 codeGraph |
| 图数据库 | Neo4j | 代码/知识关系图谱 |
| 向量数据库 | Qdrant | 语义检索与 RAG |
| 核心后端 | Go | 业务 API 与编排 |
| 图谱可视化 | D3.js + Sigma.js | 交互式图谱展示 |
| MCP 服务 | Python | 对外 Agent 接口 |
| 全文检索 | OpenSearch | 文档与代码全文搜索 |
| 关系型数据库 | MySQL | 业务持久化 |
| 前端框架 | Next.js 14 + React 18 | Web 应用 |
| UI 组件库 | Ant Design | 企业级 UI |
| API 层 | Apollo GraphQL | 前后端契约 |
| ORM | Objection.js | 数据访问（Node/JS 侧） |
| 缓存 + 任务队列 | Redis + Celery | 缓存与异步任务 |
| AI 编排 | LangChain | LLM 工作流 |
| 监控 | Langfuse | LLM 可观测性 |
| 容器化 | Docker + Nginx | 部署与反向代理 |

**跨语言协作原则：**

- 各语言模块通过**明确的 API 契约**（GraphQL Schema、REST、gRPC、消息队列）通信，禁止隐式耦合
- Rust 侧专注高性能解析与图谱构建；Go 侧承载核心业务；Python 侧承载 MCP 与 AI 编排
- 新增依赖或框架须与上表技术栈对齐，不得擅自引入替代方案

## AI Agent 角色定义

在本仓库中，AI Agent 的角色是**「企业研发协作代理」**，而非「代码补全工具」。

### 必须做到

- **先理解上下文，再动手修改**
- **优先保证业务正确性、兼容性、可维护性**
- **给出完整可审查的改动，而不是半成品**
- **能补测试时补测试，能验证时做验证**
- **如存在风险、假设、未验证项，必须明确说明**

### 禁止事项

- **未经说明直接引入破坏性变更**
- **偷改外部接口、事件结构、数据库字段语义**
- **把业务逻辑塞进 API、ORM 层**
- **为了「看起来更优雅」做大面积无关重构**
- **虚构测试通过结果、运行结果、压测结果**
- **绕过安全、权限、审计、日志、监控要求**

### 工作方式

1. **探索优先**：修改前先阅读相关代码、配置与已有测试；使用 codegraph / 项目索引理解调用链与影响面
2. **最小闭环**：一次改动解决一个明确问题，附带必要验证
3. **显式沟通**：对破坏性变更、迁移步骤、回滚方案提前说明
4. **尊重分层**：业务逻辑放在应用/领域层，API 层只做协议适配与校验

## 分层职责

| 层 | 职责 | 禁止 |
|----|------|------|
| **API / GraphQL** | 请求校验、鉴权、DTO 映射、错误码转换 | 业务规则、事务编排 |
| **应用 / 服务层** | 业务编排、领域规则、事务边界 | 直接操作 HTTP/GraphQL 细节 |
| **领域 / 模型层** | 实体、值对象、领域事件 | 依赖框架或基础设施 |
| **仓储 / 持久化** | CRUD、查询、映射 | 业务决策 |
| **基础设施** | DB、缓存、消息、外部 API、日志 | 业务逻辑 |

**前端（Next.js + Ant Design）：**

- 页面组件负责展示与交互；数据获取通过 GraphQL Client 或统一 API 层
- 不在 UI 组件中硬编码业务规则；共享逻辑抽取为 hooks 或 utils
- 图谱可视化（D3.js / Sigma.js）与业务状态解耦

**MCP 服务（Python）：**

- MCP 工具暴露平台能力，不重复实现 Go 后端已有逻辑
- 调用后端 API 或共享服务，保持权限与审计一致

## 可观测性规则

凡是会影响生产行为的改动，AI 都必须考虑可观测性。至少要检查：

- 失败时是否有可定位日志
- 是否带业务标识、traceId、请求标识
- 外部调用是否可追踪
- 关键业务动作是否可度量
- 日志是否泄露敏感信息

如果仓库已存在统一日志、埋点、Tracing 标准，**必须复用，不得另起一套**。

**LLM 相关改动**额外检查 Langfuse 埋点是否完整（prompt、token、latency、错误）。

---

## 测试规则

AI 必须按改动风险补测试。最低要求：

| 改动类型 | 最低测试要求 |
|----------|--------------|
| 应用编排修改 | 应用服务测试 |
| 接口行为修改 | 接口测试或契约测试 |
| 持久化逻辑修改 | 仓储或集成测试 |
| Bug 修复 | 尽量补回归测试 |

如果环境限制无法补测试，**必须在结果说明中明确写出原因**。

## 变更范围控制

AI 必须以**「最小闭环交付」**为原则。

### 允许

- 为完成需求进行必要的小范围抽取和整理
- 为提升可读性抽私有方法
- 与现有风格保持一致的小范围整理

### 默认不允许

- 大面积重命名
- 横跨多个模块的无关重构
- 风格化批量格式调整
- 引入新框架替代旧框架
- 没有业务收益的大规模「架构优化」

---

## 安全与权限

- **认证与授权**：所有 API、MCP 工具、GraphQL 字段须遵循现有 RBAC/ABAC 模型，不得绕过
- **敏感数据**：密钥、Token、PII 不得写入日志、测试夹具或提交历史
- **代码同步**：企业代码库同步涉及源码访问，须遵循权限边界与审计要求
- **MCP 对外暴露**：仅暴露必要能力，输入输出须校验，防止注入与越权
- **依赖安全**：引入新依赖前评估许可证与安全漏洞

---

## 数据与存储约定

- **MySQL**：业务实体与事务性数据；Schema 变更须附带迁移脚本与回滚说明
- **Neo4j**：代码/知识关系图谱；节点与边的语义变更须评估下游查询与可视化影响
- **Qdrant**：向量索引；embedding 模型或维度变更须考虑重建策略
- **OpenSearch**：全文索引；mapping 变更须评估重建与兼容性
- **Redis**：缓存与 Celery 队列；Key 命名遵循项目既有规范

**禁止**未经说明修改字段语义、删除字段或变更事件结构。

---

## Git 与提交规范

- 仅在用户明确要求时创建 commit；不擅自 push
- Commit message 聚焦「为什么」，使用完整句子
- 不提交 `.env`、凭证、密钥等敏感文件
- PR 描述须包含：变更摘要、验证方式、风险与假设

---

## 最终输出模板

AI 完成任务后，输出说明**至少**应包含：

```markdown
## 变更摘要

### 修改了哪些文件
- `path/to/file` — 简要说明

### 改了什么行为
[描述用户可感知或系统可观测的行为变化]

### 为什么这样改
[设计决策与取舍]

### 做了哪些验证
- [ ] 单元测试：`命令与结果`
- [ ] 集成/接口测试：`命令与结果`
- [ ] 手动验证：`步骤与结果`
- [ ] Lint / 类型检查：`命令与结果`

### 有哪些假设
- [列出未在代码中验证的假设]

### 还剩哪些风险或未验证项
- [列出已知风险、边界条件、待跟进项]
```

## 参考规则

Cursor 会在匹配文件时自动加载 `.cursor/rules/` 下的规则。各规则是本文档的**领域补充**，更具体时不与本文件冲突；如有冲突，以本文档为准。

| 规则文件 | 适用场景 | 触发方式 |
|----------|----------|----------|
| [frontend.mdc](.cursor/rules/frontend.mdc) | React/Next.js 组件、Apollo GraphQL、Ant Design、前端性能 | 匹配 `**/*.{tsx,jsx}` 及 `frontend/`、`web/` 目录 |
| [backend-api.mdc](.cursor/rules/backend-api.mdc) | GraphQL Resolver、Service/UseCase 编排、LangChain/Qdrant/Redis 调用 | 匹配 `backend/`、`api/`、`graphql/`、`resolvers/`、`services/` |
| [database.mdc](.cursor/rules/database.mdc) | Objection.js 模型、Knex 迁移、Repository 查询、事务与一致性 | 匹配 `models/`、`migrations/`、`repositories/`、`db/` |
| [testing.mdc](.cursor/rules/testing.mdc) | 单元/集成/契约测试选型、Mock 策略、回归测试 | 匹配 `**/*.{test,spec}.*`、`__tests__/`、`tests/` |
| [observability.mdc](.cursor/rules/observability.mdc) | 结构化日志、traceId、Langfuse 埋点、敏感信息屏蔽 | 匹配 `**/*.{ts,tsx,js,go,py}` |
| [change-control.mdc](.cursor/rules/change-control.mdc) | 变更范围控制、PR 自检与代码评审清单 | **始终生效**（`alwaysApply: true`） |

> 每条规则均以「**AI 决策前应首先检查本规则**」开头，并包含必须做/禁止做清单与正反面代码示例。
