### 改了什么行为

- **仅数据层**：`docker compose up -d`（与 Batch 0 一致）
- **全栈部署**：`docker compose --profile app up -d --build`
  - 自动迁移 + seed
  - Nginx 统一入口 `:8080`
  - 路由：`/graphql`、`/api/chat/*`、`/mcp` + 三前端（localhost 与子域）

### 为什么这样改

- **`--profile app`**：本地开发仍可按需只启数据层，全栈按需构建
- **Nginx resolver 模式**：Docker 内 DNS 延迟解析，避免 upstream 在 `nginx -t` / 启动时解析失败
- **migrate 一次性任务**：保证 API 启动前 schema 与种子数据就绪

## 根因分析（有运行时证据）

| 假设 | 结论 | 证据 |
|------|------|------|
| **H1** `proxy.golang.org` 在 Docker 构建网络中不可达/超时 | **已确认** | 默认构建：`dial tcp 142.250.69.177:443: i/o timeout`（约 30s） |
| **H2** Docker DNS 解析失败 | **已排除** | 同一容器内 `goproxy.cn` 可正常访问 |
| **H3** `go.sum` 不完整导致额外拉取 | **已排除** | 切换 GOPROXY 后所有模块均 `200 OK` |
| **H4** 需设置国内 Go 模块镜像 | **已确认** | `GOPROXY=https://goproxy.cn,direct` 后构建成功（约 4s） |

**结论**：`Dockerfile.core` 未设置 `GOPROXY`，Go 默认走 `proxy.golang.org`，在你当前网络环境下会超时。
