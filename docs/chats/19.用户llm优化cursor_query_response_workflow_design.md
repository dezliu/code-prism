### 改了什么行为

用户提问后的完整链路：

```
SecurityGuard → LoadContext → TemplateMatch → IntentClassify
  → (Clarify | DirectAnswer | Refuse | RAG 路径)
  → RagPrepare → RagRetrieve（意图分路 + hybrid/graph）→ RagQualityGate
  → (HyDE 扩写循环 | TemplateApply | GenerateAnswer)
  → GroundingCheck → StreamOutput
```

- **模板双路径**：score ≥ 0.85 自动套用结构化输出；0.6~0.85 推送 hint 卡片
- **检索质量**：Core 使用 embedding + OpenSearch BM25 + Qdrant 向量，RRF 融合
- **架构问题**：走 Neo4j 子图 + 混合检索
- **RBAC**：会话上下文注入 `allowedRepoIds`，检索时过滤
- **MCP**：配置 `MCP_SERVICE_TOKEN` 后走 `/internal/mcp/ask`，会话可持久化

### 为什么这样改

参考 nl-hermes 的**确定性状态机 + 多层质量门控**模式，而非 Agent 自由循环；在现有 SSE 骨架上最小侵入扩展，保留 API 契约兼容。

### 有哪些假设

- 配置 `ZHIPU_API_KEY` 后 embedding 与 LLM 走真实 API；未配置时 hash/placeholder 降级
- OpenSearch 可用时自动建索引；不可用时仅向量 + MySQL LIKE
- Neo4j 已有 `GraphNode`/`REL` 图谱数据（索引阶段写入）
- MCP 持久化需 API 侧配置 `MCP_SERVICE_TOKEN`

### 还剩哪些风险或未验证项

- Qdrant 从 hash 向量迁移到真实 embedding 需对已有数据执行 reindex（`scripts/reindex_knowledge.py --doc-id <id>`）
- `people` 意图暂返回 Clarify（组织架构数据源未接入）
- GroundingCheck 增加一次 LLM 调用，延迟约 +2~4s
- 生产环境需确认 `CORE_HTTP_URL` / `MCP_SERVICE_TOKEN` / OpenSearch URL 配置正确
